当前方案

  1. react native for web。

逻辑：通过webpack,babel等别名。将native组件一一转为web组件。
问题是：每个组件需要一一对应实现，工作量大。一旦出现一个新的组件就需要重新写。版本更新慢，目前只支持React Native 0.55。当前RN版本已经0.6，版本差距很大。
目前普遍采用的一种方法。

  1. reactxp
    git
  2. 自己定义了一套DSL
  1. 转为各自平台的代码
    比如Button web和native各自实现了一次,下面是web的版本

下面是native的实现

可以考虑，问题可能是组件太少

  1. taro
    自己实现了一套View,命名规则遵循React Native

思路：可以用alias,将’react’,’react-native’改为‘@tarojs/components’。这个是for web的做法，可以借鉴。关键在于View->div的转换支持程度。

问题：View->div.事件怎么办，属性怎么办，还有TexT
解答：主要是继承了Nerv.Component这个库虚拟dom的ts库，RN支持非常薄弱，几乎可以说没有,
所以RN->web这条路走不通
例如View 里面的onLayout这个回调函数就没有。reactxp有。

Q: react native for web。怎么处理View的。也是自己实现了一套，而不是直接View->div。

走不通，rn支持太差。比如Animated这个类就没有。

View->div是实现路径。 View 继承Component，里面的render方法返回了

xxx
.然后属性设置，调用dom创建。也就是说View,Text组件等自己实现了一个方式。

  1. 动态编译
    不像taro,而是通过 RN AST-> React AST，然后用react dom 的代码生成。
    关键：找到两个AST的异同。transpile多次。

  2. EXPO也有个例子，用forweb.也可以研究下。

结论

  1. reactxp这种目录格式和组织方式是我们未来统一3端的可以参考下。
  2. 思路1。 react-native -> reactxp -> web.比如 view->RX.View->div。然后利用Rx的for web能力再在web上跑起来。也就是RN origin DSL->middle DSL(support for web)->ReactDom DSL这样的思路。这样就避免了自己直接RN DSL->React DOM DSL.工作量只要少了一大堆。而且不用碰源码。这个思路可能走不通，reactxp的组件太少。
  3. 思路2 。 react-native-for-web. 直接用别名。

    ）->div

  1. 带注释
  2. 字符串通过”|”分割符，当做数组用
  3. value为基本类型，没有object。

在实际过程中发现很多奇怪的用法:

  1. 用value当做注释，而不用comment

  2. 用/做数组分割符，而不是|。

除了上述错误类型之外，还有其他错误类型。于是决定写一个自定义的json parser来规范输入问题。总的分为词法分析和语法分析两部分。

这里 json可以推导出value, value又可以推导出Number和’true’。Number又可以推导出其它，而’true’这种是基本数据类型无法再推导其他了。对于上面这种可以推导出其他的比如json,value,Number我们就叫做非终止符nonterminal，’true’这种就叫做终止符terminal。对于Number和String右边，由于只是字符的范围限定，我们也当做terminal来处理。

因为，将上面的语法定义转为具体代码，规则如下：

  1. 如果是nonterminal，则对应转成函数
  2. terminal。 匹配当前的token类型是terminal类型，然后指针移到下一个
  3. |。If或者switch
  4. *或者+。while或者for循环
  5. ？。 if

所以左边的value,Number,json等都是函数，而右边的比如”{“,”true”都是先匹配当前token类型，然后获取下一个token。
