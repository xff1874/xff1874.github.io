react native to web

Aug 27, 2019

需要处理

  1. react native jsx组件

TouchableWithoutFeedback需要处理，onPress事件需要处理。

  1. 逻辑性的

无法转化，必须补上

  1. cssInJS

当前方案

  1. react native for web。

逻辑：通过webpack,babel等别名。将native组件一一转为web组件。
问题是：每个组件需要一一对应实现，工作量大。一旦出现一个新的组件就需要重新写。版本更新慢，目前只支持React Native 0.55。当前RN版本已经0.6，版本差距很大。
目前普遍采用的一种方法。

  1. reactxp
    git
  2. 自己定义了一套DSL
  1. 转为各自平台的代码
    比如Button web和native各自实现了一次,下面是web的版本

下面是native的实现

  1. taro
    自己实现了一套View,命名规则遵循React Native

思路：可以用alias,将’react’,’react-native’改为‘@tarojs/components’。这个是for web的做法，可以借鉴。关键在于View->div的转换支持程度。

问题：View->div.事件怎么办，属性怎么办，还有TexT
解答：主要是继承了Nerv.Component这个库虚拟dom的ts库，RN支持非常薄弱，几乎可以说没有,
所以RN->web这条路走不通
例如View 里面的onLayout这个回调函数就没有。reactxp有。

Q: react native for web。怎么处理View的。也是自己实现了一套，而不是直接View->div。

结论

  1. reactxp这种目录格式和组织方式是我们未来统一3端的可以参考下。
  2. 思路。 react-native转为reactxp.比如 view->RX.View。然后利用Rx的for web能力再在web上跑起来。也就是RN origin DSL->middle DSL(support for web)->ReactDom DSL这样的思路。这样就避免了自己直接RN DSL->React DOM DSL.工作量只要少了一大堆。而且不用碰源码。

从零实现BI-JSON-Parser

Aug 26, 2019

简介

自从埋点平台上线之后，格式问题日益严重。我们默认的BI埋点如下:

一种自定义json格式，比较不同在于：

  1. 带评论注释
  2. 字符串通过”|”分割符，当做数组用
  3. value为基本类型，没有object。

在实际过程中发现很多奇怪的用法:

  1. 不用comment，用value当注解的

  2. 用其他比如||或者/做分割符，或者混用的。

错误类型很多，必须有一种方法来保证格式的规范性，所以决定一个json parser来处理上面的问题。

编译过程一般有6步。

  1. 词法解析(Lexical Analysis): 将源码转为一串的token
  2. 语法解析(Syntax Analysis): 将一串的token根据语法规则转为抽象语法树(AST)
  3. 语义解析(Semantic Analysis): 类型检查等。比如 a/b,b不能是字符串。
  4. 中间码(Intermediate Code Generation): 为生成最终代码做准备。
  5. 优化
  6. 代码生成(Code Generation): 指定生成特定的机器码。

我们主要是用到了词法解析和语法这两步。

定义好语法规则之后，转为代码的有以下5条规则。

  1. nonterminal。 对应转成函数
  2. terminal。 匹配当前的token类型是terminal类型，然后指针移到下一个
  3. |。If或者switch
  4. *或者+。while或者for循环
  5. ？。 if

所以根据上面的转换规则及语法。
我们将json的语法转为如下。

  1. 根据5条规则里面的第一条规则，nonterminal转为函数。所以我们就有parseJSON,parseValue,parseObject,parseString,parseNumber这几个函数。
  2. 根据规则2,如果是terminal就匹配token类型，然后指针移动返回下一个token。所以当parseValue碰到的token类型是true和false的时候，直接调用this.eat()匹配当前的token类型，获取下一个token.
